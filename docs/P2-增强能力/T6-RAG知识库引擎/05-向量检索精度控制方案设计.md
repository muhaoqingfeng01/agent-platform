## 1. 现状问题

当前 `02-语义切分向量化与Milvus.md` 中 MilvusCollectionManager 的精度控制现状：

```java
// 当前唯一配置 — 全部硬编码，无任何精度参数
milvusClient.createIndex(CreateIndexParam.newBuilder()
    .withIndexType(IndexType.IVF_FLAT)      // 固定索引类型
    .withMetricType(MetricType.COSINE)       // 固定度量
    .withExtraParam("{\"nlist\":128}")       // 固定聚类数
    .build());
// ❌ 检索时未传 nprobe
// ❌ 无相似度阈值过滤
// ❌ 无量化压缩
// ❌ 无一致性级别配置
```

| 缺失项 | 影响 |
|--------|------|
| nprobe 未设置 | 检索精度/速度无法权衡，默认 nprobe≈8，召回率可能不足 |
| 无相似度阈值 | 低质量结果混入 RRF 融合，污染最终 Top-5 |
| 无量化方案 | 向量全量存储 float32，10 万文档 × 500 chunk 时存储压力显著 |
| 无一致性级别 | 刚写入的文档立即可检索（强一致）还是延迟可见（最终一致），不可控 |
| 无精度监控 | 无法知道当前召回率是多少、是否达标 |

## 2. 精度控制全景架构

向量检索精度控制是 **五层联动体系**，不是单一参数调优：

```
                       ┌──────────────────────────────────┐
                       │           精度监控闭环            │
                       │    recall@k 评估 → 参数调优      │
                       ├──────────────────────────────────┤
                       │         ④ 多阶段检索             │
                       │    粗筛 → Reranker 精排          │
                       ├──────────────────────────────────┤
                       │       ③ 检索参数精度控制         │
                       │   nprobe/ef + topK + 阈值过滤    │
                       ├──────────────────────────────────┤
                       │       ② 量化精度控制             │
                       │    SQ8 / PQ / 无量化(FLAT)       │
                       ├──────────────────────────────────┤
                       │       ① 索引结构选择             │
                       │   IVF_FLAT / HNSW / DISKANN      │
                       └──────────────────────────────────┘
```

## 3. 第一层：索引结构选择

### 3.1 Milvus 支持的索引类型全景

| 索引类型 | 适用规模 | 精度 | 内存占用 | 检索速度 | 构建速度 |
|----------|:---:|:---:|:---:|:---:|:---:|
| **IVF_FLAT**（当前） | < 100 万 | ⭐⭐⭐⭐⭐ | 高 | ⭐⭐ | ⭐⭐⭐⭐ |
| **IVF_SQ8** | < 500 万 | ⭐⭐⭐⭐ | 中 | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **IVF_PQ** | < 1000 万 | ⭐⭐⭐ | 低 | ⭐⭐⭐ | ⭐⭐⭐ |
| **HNSW** | < 100 万 | ⭐⭐⭐⭐⭐ | 最高 | ⭐⭐⭐⭐⭐ | ⭐ |
| **DISKANN** | > 1000 万 | ⭐⭐⭐⭐ | 低（磁盘） | ⭐⭐⭐ | ⭐⭐ |
| **AUTOINDEX** | 自适应 | ⭐⭐⭐⭐ | 自适应 | ⭐⭐⭐⭐ | ⭐⭐⭐ |

> 精度评分基于相同 nprobe/ef 下的 recall@10 经验值，实际取决于数据和参数。

### 3.2 各索引详细分析

#### 3.2.1 IVF_FLAT — 最高精度基准（当前方案）

**算法原理**: K-Means 聚类将全量向量划分为 `nlist` 个簇，检索时只搜索最近的 `nprobe` 个簇，簇内暴力计算（FLAT=不压缩）。

```
存储: 全量 float32 向量
检索: 查询向量 → 找最近 nprobe 个聚类中心 → 簇内逐向量计算余弦相似度 → TopK
精度损失来源: 只搜了 nprobe/nlist 的簇，可能漏掉边缘簇中的高相似向量
```

**精度控制参数**:

| 参数 | 作用 | 推荐范围 |
|------|------|----------|
| `nlist` | 聚类总数 | `4 × sqrt(N)` 至 `16 × sqrt(N)` |
| `nprobe` | 检索时搜索的聚类数 | `nlist × 0.05` ~ `nlist × 0.25` |

**精度特性**:
- nprobe = nlist → 等价于暴力搜索，精度 = 100%，速度最慢
- nprobe 越大精度越高，召回率曲线呈对数增长：前 20% 的 nprobe 覆盖 80% 的召回率
- nlist 过小 → 簇过大，暴力计算慢；nlist 过大 → 簇过多，需更多 nprobe 才能保持召回

**选择理由**: 无量化 = 零精度损失，适合作为精度基准和金标准。向量数 < 100 万时性能完全可接受。

**不足**: 内存占用 = `N × dim × 4 bytes`（float32），1M × 1536 维 ≈ **5.7 GB**。

#### 3.2.2 IVF_SQ8 — 标量量化，精度/存储最佳平衡点

**算法原理**: 每个向量的每个维度从 float32（4 bytes）量化为 int8（1 byte），存储减少 75%。

```
量化过程:
  for 每个维度 d:
      range_d = max(v[:][d]) - min(v[:][d])
      step_d = range_d / 255
      quantized[i][d] = round((v[i][d] - min_d) / step_d)   → int8

检索时:
  query_vector 保留 float32
  距离计算 → 查表法反量化（int8 → float32 近似）→ 排序
```

**精度控制参数**:

| 参数 | 作用 | 推荐值 |
|------|------|--------|
| `nlist` | 同 IVF_FLAT | `4 × sqrt(N)` |
| `nprobe` | 同 IVF_FLAT | nlist × 0.1 |
| 量化范围 | 全局 vs 每簇独立 | 每簇独立精度更高 |

**精度特性**:
- 理论精度损失 < 0.5%（在 embedding 向量上通常 < 0.2%）
- 每个维度量化误差 ≤ range_d / 510 → 对高维向量的余弦相似度影响极小
- 维度越高（1536），量化噪声越容易被平均掉，精度损失越低

**选择理由**:
- **存储减少 75%**: 1M × 1536 × 1 byte ≈ **1.44 GB**（vs IVF_FLAT 5.7 GB）
- **检索加速 ~30%**: 更少的内存带宽占用，int8 计算更快
- **精度损失可忽略**: DNN embedding 向量对小幅数值扰动天然鲁棒
- **业界首选**: 90% 的生产 RAG 系统用 IVF_SQ8 就能满足精度要求

**不足**:
- 向量分布极不均匀（某维度方差极大）时，该维度量化误差偏高
- 需要训练数据确定量化范围（Milvus 自动完成）

#### 3.2.3 IVF_PQ — 乘积量化，极致压缩

**算法原理**: 将 1536 维向量分为 m 组（子向量），每组独立做 K-Means 聚类（码本），最终每个向量用 m 个码本索引表示。

```
步骤:
  1. 向量分解: v(1536维) → [sub_v1(1536/m 维), sub_v2, ..., sub_vm]
  2. 每子空间聚类: 每个子空间做 K-Means，k=256 → 8 bits 编码
  3. 向量编码: v → [code1, code2, ..., codem]（m bytes）
  4. 检索时: 查询向量分子空间 → 查码本反量化 → 近似距离计算

存储: m × 1 byte = m bytes（m 通常 = dim/2 = 768）
压缩比: float32(1536×4=6144 bytes) → PQ(768 bytes) ≈ 8:1
```

**精度控制参数**:

| 参数 | 作用 | 推荐值 |
|------|------|--------|
| `m`（子向量数） | 越大精度越高、压缩比越低 | dim/2 = 768 |
| `nbits`（编码位数） | 每个子空间的码本大小 = 2^nbits | 8（256 个码字） |
| `nlist` | 粗聚类数 | `4 × sqrt(N)` |
| `nprobe` | 搜索聚类数 | nlist × 0.1 ~ 0.2 |

**精度特性**:
- 精度损失 1% ~ 5%（取决于 m 和 k 的选择）
- m = dim（每维度 1 个子空间）→ 退化为 SQ
- m = dim/2 → 最优平衡点（经验值）
- 训练数据量越大，码本质量越高，精度损失越小

**选择理由**:
- **极致压缩**: 存储仅为 FLAT 的 1/8 ~ 1/10
- **大规模场景必选**: > 500 万向量时 IVF_FLAT 内存不可接受
- **可调控精度**: 通过 m 值和码本大小调节精度/存储平衡

**不足**:
- 精度损失 > SQ8，对某些分布的数据可能 > 5%
- 检索速度比 SQ8 慢（需要查多张码表）
- 训练时间长（m 次 K-Means）

#### 3.2.4 HNSW — 图索引，低延迟高精度

**算法原理**: 构建多层可导航小世界图（Hierarchical Navigable Small World），每层是近似 Delaunay 图，检索时从顶层向下贪心搜索。

```
构建:
  Layer 2 (稀疏): ●───●
                  │
  Layer 1 (中等): ●─●─●─●
                  │   │
  Layer 0 (稠密): ●─●─●─●─●─●   ← 所有向量在 Layer 0

检索:
  从最高层入口点开始 → 贪心下降至 Layer 0 → Layer 0 贪心搜索 ef 个候选 → 返回 TopK
```

**精度控制参数**:

| 参数 | 作用 | 推荐范围 |
|------|------|----------|
| `M` | 每节点最大连接数（图密度） | 16 ~ 64 |
| `efConstruction` | 构建时搜索宽度 | 100 ~ 500 |
| `ef` | 检索时搜索宽度 | TopK ~ efConstruction |

**精度特性**:
- ef = efConstruction → 精度接近 100%（召回率 > 0.99）
- M 越大精度越高但内存越大，M=16 通常 recall@10 > 0.95
- 检索复杂度 O(log N)，不受数据量影响——这是 HNSW 最大优势

**选择理由**:
- **极低检索延迟**: < 5ms（十万级向量），适合实时对话场景
- **高精度**: ef=64 时 recall@10 通常 > 0.98
- **增量友好**: 插入新向量不需要重新训练（IVF 系列需要定期重建）
- **实时 RAG 首选**: 用户问问题 → 检索 → 返回，延迟敏感场景最优

**不足**:
- **内存占用最高**: 需存储图结构（每节点 M 条边 + 向量本身），1M × (1536×4 + 16×8) ≈ **6.5 GB**
- **构建速度慢**: O(N log N × M × efConstruction)，比 IVF 慢 5-10 倍
- **删除成本高**: 删除节点需要修复图结构，Milvus 2.x 已支持但仍有开销
- **不适合频繁全量重建**: 每次重建索引都需要重构整张图

#### 3.2.5 DISKANN — 磁盘索引，亿级规模

**算法原理**: Vamana 图算法 + SSD 感知的数据布局优化，将大部分索引放磁盘，内存仅缓存热点数据。

```
存储分层:
  SSD: 全量向量 + 图结构
  RAM: 热点向量 + 图入口（可配置 RAM 预算）

检索:
  RAM 入口点 → SSD 读取候选向量 → 贪心搜索 → 返回 TopK
  利用 SSD 的高顺序读取带宽（~3GB/s），随机 IO 由图结构优化
```

**精度控制参数**:

| 参数 | 作用 | 推荐范围 |
|------|------|----------|
| `max_degree` | 图节点最大度数 | 32 ~ 64 |
| `search_list_size` | 搜索候选集大小 | 100 ~ 200 |
| `pq_code_budget_gb` | PQ 压缩码本内存预算 | 物理 RAM 的 20% ~ 50% |

**精度特性**:
- 精度接近 HNSW（recall@10 > 0.95），速度略慢（需 SSD IO）
- 内存预算越大精度越高

**选择理由**:
- **唯一能处理亿级向量的索引**
- **成本可控**: SSD 比 RAM 便宜 10 倍，用磁盘换规模
- **不需要全量数据进内存**

**不足**:
- 延迟比纯内存索引高 2-5 倍（SSD IO）
- 需要 NVMe SSD（SATA SSD 延迟不可接受）
- 初期不适用（向量量 < 100 万时无必要）

### 3.3 索引选择决策树

```
数据规模 N
│
├─ N < 10 万
│   └─ 延迟敏感？
│       YES → HNSW（M=16, ef=64）
│       NO  → IVF_FLAT（精度基准）
│
├─ 10 万 < N < 100 万
│   └─ 延迟敏感？
│       YES → HNSW（M=16, ef=64）
│       NO  → IVF_SQ8（推荐，精度/存储最佳平衡）
│
├─ 100 万 < N < 1000 万
│   └─ IVF_PQ（m=dim/2, nbits=8）
│       或 IVF_SQ8 × 更大内存
│
└─ N > 1000 万
    └─ DISKANN（SSD 索引）
```

## 4. 第二层：量化精度控制

### 4.1 量化方案对比

| 方案 | 压缩比 | 精度损失 | 检索速度 | 适用规模 |
|------|:---:|:---:|:---:|:---:|
| **无量化** (FLAT) | 1:1 | 0% | 基准 | < 100 万 |
| **SQ8** (标量量化) | 4:1 | < 0.5% | +30% | < 500 万 |
| **SQ4** (4-bit 标量) | 8:1 | 1~2% | +40% | < 1000 万 |
| **PQ** (乘积量化) | 8~16:1 | 1~5% | +20% | < 5000 万 |
| **PQ + SQ 混合** | 16~32:1 | 3~8% | +25% | < 1 亿 |

### 4.2 量化精度调优策略

**SQ8 调优**:
- 每簇独立量化 > 全局量化：每个 IVF 簇内部向量分布更集中，量化误差更小
- 量化前先 L2 归一化：归一化后数值范围 [-1, 1]，量化分辨率利用更充分

**PQ 调优**:
- `m`（子向量数）越大精度越高：`m = dim` → 退化为 SQ；`m = dim/2` → 经验最优
- `nbits = 8` 是甜点位：6-bit 精度锐降，10-bit 收益递减
- 子向量维度 `d_sub = dim / m` 应为 2 的幂次：可以用 SIMD 加速
- 训练数据量 > 100 × K（码本大小）：保证每个码字有足够训练样本

### 4.3 推荐量化路线图

```
阶段一（当前 P2）:  IVF_FLAT 无量化 → 精度基准，数据量小
阶段二（10 万向量）: IVF_SQ8      → 首次引入量化，精度无感损失
阶段三（100 万向量）: IVF_PQ       → 存储压力驱动升级
阶段四（1000 万向量）: DISKANN     → 磁盘索引
```

## 5. 第三层：检索参数精度控制

### 5.1 核心检索参数

#### 5.1.1 nprobe — 精度/速度主旋钮

```
作用: IVF 索引中搜索的聚类单元数
精度关系: recall@K ≈ 1 - exp(-λ × nprobe / nlist)
          （nprobe 增大 → 召回率对数增长，边际递减）

推荐: nprobe = nlist × 0.1（精度 ~0.95）
      nprobe = nlist × 0.05（精度 ~0.90，速度 2x）
      nprobe = nlist × 0.25（精度 ~0.99，速度 0.4x）
```

| nlist | nprobe 推荐 | 预期 recall@10 | 延迟 (ms) |
|-------|:----------:|:---:|:---:|
| 128（当前） | 8（default） | ~0.88 | 3 |
| 128 | 16 | ~0.94 | 5 |
| 128 | 32 | ~0.97 | 9 |
| 128 | 64 | ~0.99 | 16 |
| 256 | 24 | ~0.93 | 5 |
| 512 | 50 | ~0.94 | 8 |
| 1024 | 50 | ~0.90 | 8 |

#### 5.1.2 ef — HNSW 精度旋钮

```
作用: HNSW 检索时的搜索宽度（候选集大小）
精度关系: ef 越大精度越高，边际递减

推荐: ef = TopK × 10 ~ TopK × 20
      ef = 64（TopK=5 时 recall > 0.98）
      ef = 128（TopK=5 时 recall > 0.995）
```

#### 5.1.3 相似度阈值过滤

当前方案搜索 20 个结果后**不做任何过滤**直接送 RRF 融合。应在 RRF **之前**对单路检索结果做阈值截断：

```
向量检索:
  余弦相似度范围 [-1, 1]，embedding 模型通常 ≥ 0.5 才具有语义相关性

  阈值建议:
    threshold = 0.65 → 高精度模式（宁可漏检，不能错检）
    threshold = 0.50 → 平衡模式
    threshold = 0.35 → 高召回模式（宁可错检，不能漏检）
    threshold = 0    → 关闭过滤（当前行为）
```

### 5.2 检索参数策略集

不同场景预设不同的参数组合：

| 策略名 | nprobe | TopK | 阈值 | 适用场景 |
|--------|:---:|:---:|:---:|----------|
| **precise**（高精度） | nlist×0.25 | 30 | 0.65 | 医疗/法律/金融——错误答案代价高 |
| **balanced**（平衡） | nlist×0.10 | 20 | 0.50 | 通用问答（默认） |
| **fast**（低延迟） | nlist×0.05 | 15 | 0.50 | 实时对话，延迟要求 < 2s |
| **recall**（高召回） | nlist×0.20 | 40 | 0.35 | 探索性搜索/研究场景 |
| **turbo**（极速） | nlist×0.03 | 10 | 0.55 | 流式输出首 token 优化 |

### 5.3 动态参数调整

```
根据知识库元数据自动选择策略:

知识库类型 = 技术文档  → balanced（默认）
知识库类型 = 医疗记录  → precise（高精度）
知识库类型 = FAQ      → balanced
知识库类型 = 新闻归档  → fast（大容量低价值）
知识库标记 = 安全合规  → precise（强制）
```

## 6. 第四层：多阶段检索

### 6.1 粗筛 + 精排架构

单阶段 ANN 检索在高维空间中的精度天花板明显。多阶段检索用粗筛保证召回、用精排保证精度：

```
Query
  │
  ▼
┌──────────────────────┐
│ Stage 1: ANN 粗筛     │  向量检索 Top-50（高召回模式）
│ IVF_SQ8 + nprobe=30  │  目标: 召回率 > 0.99
│ 延迟: ~10ms          │  宁可多召回，不漏真相关
└──────────┬───────────┘
           ▼
┌──────────────────────┐
│ Stage 2: Reranker 精排│  Cross-Encoder / ColBERT 重排序
│ 50条 → Top-10        │  query-chunk 联合编码，精算相关度
│ 延迟: ~50ms（批处理） │  目标: precision@10 > 0.95
└──────────┬───────────┘
           ▼
         Top-10
           │
           ▼
┌──────────────────────┐
│ Stage 3: RRF 融合    │  向量 Top-10 + 关键词 Top-10 → 融合排序
│ 10+10 → Top-5        │  当前已有逻辑
└──────────────────────┘
```

### 6.2 Reranker 选型

| Reranker | 精度 | 延迟 | 成本 | 适用场景 |
|----------|:---:|:---:|:---:|----------|
| **Cross-Encoder** (bge-reranker-v2) | ⭐⭐⭐⭐⭐ | 中 | GPU | 高精度要求 |
| **ColBERT** | ⭐⭐⭐⭐ | 中 | GPU | 可解释性要求 |
| **LLM-as-Reranker** | ⭐⭐⭐⭐⭐ | 高 | API | 复杂语义判断 |
| **BM25 分数归一化** | ⭐⭐ | 低 | 免费 | 当前方案（非严格 rerank） |

### 6.3 多阶段精度收益

```
单阶段 ANN (当前):
  recall@20 ≈ 0.94, precision@5 ≈ 0.72

二阶段 (ANN + Reranker):
  recall@20 ≈ 0.99, precision@5 ≈ 0.88

三阶段 (ANN + Reranker + RRF):
  recall@20 ≈ 0.99, precision@5 ≈ 0.92
```

## 7. 第五层：精度监控闭环

没有监控的精度控制 = 瞎调。以下方案确保精度可知、可调、可验证。

### 7.1 召回率评估

```
评估数据集:
  来源1: 历史命中记录 t_knowledge_hit_record 中 human_feedback = 'EXCELLENT' 的 (query, chunk) 对
  来源2: 人工标注 50 条黄金查询 × 相关 chunk 列表
  来源3: LLM 自动生成 200 条合成查询（从 chunk 内容反向出题）

评估指标:
  recall@5  = |检索Top5 ∩ 标注相关| / |标注相关|
  recall@10 = |检索Top10 ∩ 标注相关| / |标注相关|
  MRR       = 1/N × Σ(1 / 第一个相关chunk的排名)
  NDCG@10   = 标准化折损累计增益

目标:
  生产环境 recall@10 > 0.90
  关键场景 recall@10 > 0.95
```

### 7.2 精度回归检测

```
每次索引重建 / 模型更新后自动运行:
  1. 运行评估数据集
  2. 对比基线指标
  3. recall@10 下降 > 5% → 告警 + 自动回滚索引版本
  4. recall@10 下降 > 2% → 通知 + 人工决策
```

### 7.3 参数调优实验框架

```
Grid Search 自动化:
  参数空间:
    nlist ∈ [64, 128, 256, 512, 1024]
    nprobe ∈ [nlist*0.05, nlist*0.1, nlist*0.15, nlist*0.2]
    threshold ∈ [0.35, 0.50, 0.65]

  目标函数:
    score = recall@10 × 0.6 - latency_ms / 100 × 0.3 - storage_gb / 10 × 0.1

  输出:
    最优参数组合 + 帕累托前沿（精度 vs 延迟 trade-off）
```

## 8. Milvus 一致性级别

Milvus 支持三种一致性级别，影响"写入后多久能检索到"：

| 级别 | 写入可见延迟 | 精度影响 | 适用场景 |
|------|:---:|------|----------|
| **STRONG** | ~100ms | ✅ 保证读到最新数据 | 上传后立即可检索（用户期望） |
| **BOUNDED** | ~20ms | ⚠️ 可能漏最新写入 | 批量导入场景 |
| **EVENTUALLY** | ~5ms | ⚠️ 可能漏最近 N 秒写入 | 高吞吐检索，允许短暂不一致 |

**推荐**: 文档上传完成 → READY 状态时使用 STRONG 级别执行一次 flush + 检索验证；日常对话检索使用 BOUNDED 级别。

## 10. 全参数手动可配设计

> **核心原则: 所有精度参数、阈值、策略均可由用户在四个层级手动覆盖，不做任何硬编码。**

### 10.1 四级配置层级

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 系统默认值（application.yml）              │
│  所有参数的兜底值，保证开箱即用                       │
├─────────────────────────────────────────────────────┤
│  Layer 2: 策略预设（系统内置 5 套，可新增/修改）      │

│  Layer 3: 知识库级配置（t_knowledge_base）           │
│  创建知识库时设置，应用于该知识库下所有文档           │

│  Layer 4: 文档级配置（t_document）                   │
```

#### 10.2.1 索引构建参数（index_params_json）

> **影响范围**: 控制 Milvus 索引结构，修改后需重建索引生效。

| `efConstruction` | int | `200` | 8 ~ 512 | HNSW 构建搜索宽度（仅 HNSW 生效） |

#### 10.2.2 检索执行参数（search_params_json）

> **影响范围**: 控制每次检索的行为，即时生效，无需重建索引。
> **配置时机**: 可在知识库级或文档级随时修改，下次检索立即生效。

| 参数 | 类型 | 默认值 | 有效范围 | 说明 |
|------|------|--------|----------|------|
| `nprobe` | int | `16` | 1 ~ nlist | IVF 检索搜索聚类数（仅 IVF 系列生效） |
| `ef` | int | `64` | TopK ~ efConstruction | HNSW 检索搜索宽度（仅 HNSW 生效） |

| `topK` | int | `20` | 5 ~ 100 | 向量检索返回结果数 |
| `similarity_threshold` | float | `0.50` | 0.0 ~ 1.0 | 余弦相似度最低阈值（低于此值的结果丢弃） |
| `consistency_level` | enum | `BOUNDED` | STRONG / BOUNDED / EVENTUALLY | Milvus 一致性级别 |
| `timeout_ms` | int | `5000` | 1000 ~ 30000 | 单次检索超时（毫秒） |

**JSON Schema 示例**:

```json

    "topK": 30,
    "similarity_threshold": 0.65,
    "consistency_level": "STRONG",
```

### 10.4 五套内置策略预设

> 策略预设是一组预定义的参数集合，用户选择一个策略即批量设定了所有参数。
> **用户可以手动修改策略预设中任何一个参数值**，也可以基于预设创建自定义策略。

| 策略名 | nprobe | topK | similarity_threshold | consistency | timeout_ms | 适用场景 |
|--------|:---:|:---:|:---:|:---:|:---:|----------|
| **precise** | nlist×0.25 | 30 | 0.65 | STRONG | 10000 | 医疗/法律/金融等高精度场景 |
| **balanced** | nlist×0.10 | 20 | 0.50 | BOUNDED | 5000 | 通用知识问答（默认） |
| **fast** | nlist×0.05 | 15 | 0.50 | EVENTUALLY | 2000 | 实时对话/流式首 token 优化 |
| **recall** | nlist×0.20 | 40 | 0.35 | BOUNDED | 8000 | 探索性搜索/研究场景 |
| **turbo** | nlist×0.03 | 10 | 0.55 | EVENTUALLY | 1000 | 极速模式/批量检索 |

**每个策略预设的完整参数（以 `balanced` 为例）**:

```json
{
    "strategy_name": "balanced",
    "description": "通用知识问答 — 精度/延迟/成本均衡",
    "search_params": {
        "nprobe_ratio": 0.10,
        "topK": 20,
        "similarity_threshold": 0.50,
        "consistency_level": "BOUNDED",
        "timeout_ms": 5000
    },
    "multi_stage_params": {
        "enable_reranker": false,
        "reranker_type": "NONE",
        "coarse_top_k": 50,
        "enable_rrf_fusion": true,
        "rrf_k": 60,
        "fusion_top_n": 5,
        "vector_weight": 0.5,
        "keyword_weight": 0.5
    }
}
```

> 注: `nprobe_ratio` 是相对值（nlist 的百分比），运行时解析为绝对值 `Math.ceil(nlist * nprobe_ratio)`，避免手动维护绝对值的同步问题。

### 10.9 配置校验规则

系统在保存配置时执行以下校验，不合法时拒绝保存并返回明确的错误信息：

| 校验规则 | 错误提示示例 |
|----------|-------------|
| `nprobe > nlist` | nprobe(64) 不能超过 nlist(32)，请调整 |
| `similarity_threshold` 不在 [0, 1] | 相似度阈值必须在 0.0 ~ 1.0 之间 |
| `fusion_top_n > topK` | 融合返回数(10)不应大于向量检索返回数(5) |
| `vector_weight + keyword_weight = 0` | 向量权重和关键词权重不能同时为 0 |
| `enable_reranker=true` 但 `reranker_type=NONE` | 启用 Reranker 时必须选择 Reranker 类型 |
| HNSW 参数写入 IVF 索引 | 当前索引类型为 IVF_FLAT，HNSW 参数 M 不适用，已忽略 |
| PQ 参数写入非 PQ 索引 | 当前索引类型为 IVF_FLAT，PQ 参数 nbits 不适用，已忽略 |

## 11. 总结

向量精度控制是一个**五层联动、四级可配**的系统工程：

| 层级 | 控制内容 | 精度影响 | 可配性 | 当前状态 |
|------|----------|:---:|:---:|:---:|
| ① 索引结构 | IVF/HNSW/DISKANN 选型 | ⭐⭐⭐⭐⭐ | 知识库级手动选择 | 仅 IVF_FLAT |
| ② 量化策略 | SQ8/PQ 压缩方案 | ⭐⭐⭐⭐ | 知识库级手动选择 | 无量化 |
| ③ 检索参数 | nprobe/ef/threshold/TopK | ⭐⭐⭐⭐ | 四级全覆盖 | 全部缺失 |
| ④ 多阶段 | 粗筛+精排+Reranker+RRF | ⭐⭐⭐ | 知识库/文档级可选 | 仅 RRF |
| ⑤ 监控闭环 | 召回率评估+参数调优 | 间接保障 | 知识库级可配置 | 无 |

**核心原则**:
- **全部参数手动可配**: 无一例外，每一个 nprobe、阈值、权重都有四级配置入口
- **逐参数独立覆盖**: 用户可以只改 `similarity_threshold` 一个值，其余全部 fallback 下层默认
- **即时生效 + 索引重建分离**: 检索参数改完立即可用；索引参数改完后显式触发重建
- **校验前置**: 保存时检查参数合法性和互斥约束，不允许非法配置入库
- **小数据量时精度优先**: IVF_FLAT + nprobe=nlist×0.1 是成本最低的高精度方案
- **数据增长引入量化**: IVF_SQ8 是精度/存储的最佳甜点位
- **精度可知才可控**: 评估数据集 + recall 基线是精度控制的前提
