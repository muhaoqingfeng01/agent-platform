# 多策略文档切片方案设计

> **日期**: 2025-06-15
> **所属阶段**: P2 增强能力 → T6 RAG 知识库引擎
> **触发**: 补充切片策略设计缺陷 — 原有仅一种段落+滑动窗口方式
> **关联文档**: [[01-文档上传与解析管线]], [[02-语义切分向量化与Milvus]], [[03-混合检索与RRF融合排序]]

---

## 1. 设计目标

将原有**单一硬编码切片方式**替换为**6 种可选切片策略**，支持：

1. **上传时手动选择**切片方式（前端传入 `chunkStrategy` 参数）
2. **根据文件类型自动兜底**选择（未指定时系统按 file_type 自动匹配最优策略）
3. **策略级可配置参数**（chunkSize、overlap、分隔符等按策略定制）
4. **知识库级默认策略**（`t_knowledge_base` 预设默认值，文档级可覆盖）

## 3. 六种切片策略详解

### 3.1 段落 + 滑动窗口切片（paragraph_sliding_window）

> **适用文件**: PDF、DOCX、TXT（通用长文本文档）
> **设计理念**: 以自然段落为最小语义单元，保持段落完整性，同时通过滑动窗口保证上下文连贯。

#### 算法流程

```
Step 1: 按空行分割段落
    text.split(/\n\s*\n/)
    → [para1, para2, para3, ...]

Step 2: 段落合并/拆分
    短段落（< minParagraphTokens=100）→ 向前合并，直到 >= minParagraphTokens
    长段落（> chunkSize）→ 在句子边界（。！？\n）处断句拆分

Step 3: 组装 Chunk
    从段落序列中顺序累加，总 token <= chunkSize 时继续追加
    超出 chunkSize → 当前 chunk 封口，新段落开下一个 chunk

Step 4: 滑动窗口重叠
    chunk[i] 头部插入 chunk[i-1] 最后 overlap 个 token 的内容
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| chunk_size | 512 | 256 ~ 2048 | 目标 token 数 |
| chunk_overlap | 50 | 0 ~ chunk_size/2 | 滑动窗口重叠 token 数 |
| min_paragraph_tokens | 100 | 50 ~ 300 | 短段落合并阈值 |

#### 选择理由

- **自然段落 = 自然语义边界**: 作者用段落来组织思想，切在段落边界不会打断完整逻辑
- **滑动窗口保证上下文连贯**: chunk[i-1] 尾部 50 token 叠加到 chunk[i] 头部，检索时不丢失跨 chunk 的信息
- **短段合并保护碎片化**: 代码注释、列表项等单行段落（通常 < 50 token）会被合并，避免生成大量低质量碎片
- **长段句子边界断句**: 对于排版差、不分段的"大段文字"，在句号/问号/感叹号处断开，而非粗暴截断

#### 不足

- 对无段落结构的文本（日志、代码、数据表）效果差
- 对标题层级结构（Markdown）无感知，会丢失章节上下文
- 段落长度极度不均匀时（如 PPT 导出 PDF），chunk 质量波动大

---

### 3.2 固定长度切片（fixed_size）

> **适用文件**: 规范文档、API 参考、代码文件、CSV 数据
> **设计理念**: 严格按固定 token 窗口切分，忽略自然边界，适用于结构高度均匀的文本。

#### 算法流程

```
Step 1: 全文本 token 化
    将全文按 token 粒度切分为 token 序列

Step 2: 固定窗口滑动
    window_start = i × (chunk_size - chunk_overlap)
    window_end = window_start + chunk_size
    直接截取 token[window_start : window_end]

Step 3: 可选软边界修正
    若 enable_boundary_softening=true：
    从 window_end 向前搜索最近的句子边界（。！？\n）
    偏移不超过 max_boundary_shift=50 token 时，在该边界处截断
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| chunk_size | 512 | 128 ~ 4096 | 固定窗口大小（token） |
| chunk_overlap | 64 | 0 ~ chunk_size/2 | 窗口重叠大小 |
| enable_boundary_softening | true | bool | 是否在句子边界微调截断点 |
| max_boundary_shift | 50 | 10 ~ 100 | 边界修正最大偏移量 |

#### 选择理由

- **确定性**: 同版本代码 + 同参数 → 完全一致的切片结果，便于缓存和增量更新
- **均匀性**: 每个 chunk 大小几乎相同，embedding 质量稳定，Milvus 索引效率最优
- **API 文档/规范文档最优**: 这类文档每个段落信息密度均匀，固定窗口不会有明显语义割裂
- **代码友好**: 函数/类/方法按固定窗口截断，配合 overlap 保证交叉引用不丢失
- **CSV/日志处理**: 表格数据和日志天然是定长结构，无自然段落概念

#### 不足

- 可能在句中断开，割裂语义
- 对叙事性文本（小说、报告）不友好
- 重叠区域可能产生大量冗余 embedding 计算

### 3.3 Markdown 标题层级切片（markdown_header_aware）

> **适用文件**: .md / Markdown 文档、Tika 转换后的 HTML（含 headinglevel 信息）
> **设计理念**: 以 Markdown 标题（`#` ~ `######`）为天然切片边界，保留完整章节层级路径作为 chunk 元数据。

#### 算法流程

```
Step 1: Markdown AST 解析
    使用 commonmark-java 或 flexmark 解析 AST
    提取节点: Heading(level 1-6), Paragraph, CodeBlock, List, Table, Blockquote

Step 2: 标题路径构建
    遍历 AST，维护当前标题栈：
    H1 → ["# 概述"]
    H2 → ["# 概述", "## 安装"]
    H3 → ["# 概述", "## 安装", "### Docker"]
    每个节点记录其完整 header_path（用于写入 metadata_json）

Step 3: 按层级切块
    遇到新标题时：
    a. 当前积攒内容 token 数 >= min_chunk_tokens → 封口生成一个 chunk
    b. 当前积攒内容 < min_chunk_tokens → 与上一个 chunk 合并（兄弟节点聚合）
    c. 内容超过 max_chunk_tokens → 内部按段落进一步细分（递归降级到 paragraph_sliding_window）

Step 4: 代码块保护
    代码块（```...```）和内联代码永不切割，作为原子单元
    单个代码块过长（> max_chunk_tokens × 2）→ 按函数/方法边界切割
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| min_chunk_tokens | 256 | 128 ~ 512 | 低于此值合并到上一个 chunk |
| max_chunk_tokens | 1024 | 512 ~ 2048 | 超过此值内部再切分 |
| honor_heading_levels | [1,2,3,4] | 1-6 子集 | 哪些层级的标题作为切分边界 |
| code_block_atomic | true | bool | 代码块是否保持原子 |
| include_header_path_in_content | true | bool | 是否将 `header_path` 前缀注入 chunk 文本 |

#### 选择理由

- **标题是文档骨架**: Markdown 的标题层级是作者明确标注的结构边界，比任何启发式切法都准确
- **标题路径注入提升检索精度**: chunk 文本前缀 `# 概述 > ## 安装 > ### Docker` → embedding 携带了章节上下文，用户查询 "Docker 怎么装" 能准确命中
- **代码块原子保护**: 代码示例不跨 chunk 切割，避免 `for` 循环在 chunk A、`}` 在 chunk B 的尴尬
- **兄弟节点智能聚合**: 同级短章节（如 API 参数列表的每个参数说明）自动合并，避免过度碎片化
- **项目文档最优方案**:技术文档 / README / Wiki 天然 Markdown 编写，此策略为最佳匹配

#### 不足

- 依赖 Markdown 写得好（标题层级规范），乱写的 .md 效果退化
- 非 Markdown 文件（PDF/DOCX）需前置 HTML→MD 转换，增加处理链路
- AST 解析比纯文本 split 慢一个数量级

### 3.4 句子级切片（sentence_level）

> **适用文件**: FAQ 文档、对话记录、客服话术、短问答对
> **设计理念**: 以完整句子为单位组装 chunk，严格控制每个 chunk 内的句子数，避免问句和答句被切散。

#### 算法流程

```
Step 1: 句子分割
    使用 OpenNLP / Stanford NLP SentenceDetector 分句
    中文增加 jieba + 正则断句（。！？；\n）
    输出: [sent1, sent2, sent3, ...]

Step 2: 句子组装
    从 sent[0] 开始累加 token 计数
    累加到 sent[k] 时 total_tokens > chunk_size → 生成 chunk[0] = sent[0..k-1]
    从 sent[k] 开始下一个 chunk
    每个 chunk 开头可选回退 overlap_sentences 个句子做重叠

Step 3: 问句-答句保护（可选）
    若 detect_qa_pairs=true：
    识别 `?` / `？` / `Q:` 结尾的句子 → 必须与紧随的回答句子放在同一 chunk 内
    识别 `A:` 开头的句子 → 强制与前一句（问句）绑定
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| chunk_size | 512 | 256 ~ 1024 | 目标 token 数（句子累加上限） |
| overlap_sentences | 2 | 0 ~ 5 | 重叠句子数 |
| detect_qa_pairs | true | bool | 是否检测 QA 对并保持完整 |
| max_sentences_per_chunk | 15 | 5 ~ 30 | 单 chunk 句子数上限（防止巨型句子） |

#### 选择理由

- **FAQ 场景最优**: FAQ 文档本质是一组（问，答）对，句子级切分不会把 "如何重置密码？" 和 "请依次点击设置→安全→重置密码" 拆到不同 chunk
- **对话场景天然匹配**: 对话记录按句子边界切分，chunk 内保持完整的表达单元
- **embedding 语义聚焦**: 每个 chunk 包含 5-15 个语义完整的句子，向量表达更精确
- **检索粒度恰当**: 用户查询通常是 1-3 句话，句子级 chunk 长度与查询长度匹配，余弦相似度计算更可靠
- **QA 对保护**: 开启后自动识别问答对并绑定，防止检索到"只有问题没有答案"的无用 chunk

#### 不足

- 强依赖分句准确率（中文分句比英文难）
- 对叙事性长文本（小说）会生成大量 chunk，且割裂叙事流
- 大段说明文字（无明确句子边界）分句效果差

### 3.5 递归字符分割（recursive_char_split）

> **适用文件**: 无结构纯文本、日志文件、混合格式文档、未识别文件类型的兜底
> **设计理念**: 按优先级递减的分隔符集递归切割——先尝试最粗粒度的分隔符，无法达到目标大小时降级到更细粒度分隔符。**这是 LangChain 默认策略的增强版**。

#### 算法流程

```
Step 1: 分隔符优先级定义
    separators = [
        "\n\n",       // L1: 段落边界（最优）
        "\n",         // L2: 行边界
        "。",         // L3: 中文句号
        ". ",         // L4: 英文句号
        "！", "？",   // L5: 中文感叹/问号
        "! ", "? ",   // L6: 英文感叹/问号
        "；", ";",    // L7: 分号
        "，", ",",    // L8: 逗号
        " ",          // L9: 空格（最后手段）
        ""            // L10: 逐字符切割（终极降级）
    ]

Step 2: 递归切割函数
    split_recursive(text, separators, chunk_size, chunk_overlap):
       当前分隔符 = separators[0]
       片段 = text.split(当前分隔符)    // 按当前分隔符切割

       chunks = []
       current_chunk = ""
       for 片段中的每个 seg:
           if token_count(current_chunk + seg) <= chunk_size:
               current_chunk += seg
           else:
               if token_count(seg) > chunk_size:
                   // 单段超限 → 递归降级到下一级分隔符
                   sub_chunks = split_recursive(seg, separators[1:], ...)
                   chunks.extend(sub_chunks)
               chunks.append(current_chunk)
               current_chunk = seg
       return chunks

Step 3: 重叠处理
    每个 chunk[i] 尾部 overlap 个 token 与 chunk[i+1] 头部重复
    overlap 边界在最近的低优先级分隔符处微调
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| chunk_size | 800 | 256 ~ 2048 | 目标 token 数 |
| chunk_overlap | 100 | 0 ~ chunk_size/2 | 重叠 token 数 |
| separator_priority_list | (见上述) | 自定义分隔符列表 | 切割优先级链 |

#### 选择理由

- **通用性最强**: 不依赖任何文档结构假设，任何纯文本都能切
- **渐进降级保证成功率**: 从段落→句子→短语→字符，永远能找到合适的切分点
- **LangChain 社区验证**: 此策略是 LangChain `RecursiveCharacterTextSplitter` 的核心算法，经海量生产环境验证
- **兜底方案不二之选**: 当文件类型无法识别、其他策略失败时，此策略 100% 可用
- **分隔符优先级可定制**: 可根据语言（中文用 `。`，英文用 `. `）调整优先级

#### 不足

- 完全忽略文档语义结构，纯启发式
- 可能在逻辑边界处断开（如列表中间）
- 最慢的切片策略之一（递归 + 多次 split）

### 3.6 语义切片（semantic）

> **适用文件**: 高质量长文（技术博客、论文、书籍章节）、追求极致检索精度的场景
> **设计理念**: 用 embedding 相似度检测语义断点——相邻句子间 embedding 相似度骤降处即为自然切片边界。这是唯一真正配得上"语义切分"名称的策略。

#### 算法流程

```
Step 1: 句子级预处理
    将全文按句子分割（同 sentence_level 的 Step 1）
    → [sent1, sent2, sent3, ..., sentN]

Step 2: 句子 embedding
    对每个句子调用 embeddingClient.embed(sent_i)
    → [vec1, vec2, vec3, ..., vecN]

Step 3: 语义断点检测
    计算相邻句子余弦相似度:
    similarity[i] = cosine(vec[i], vec[i+1])

    for i in range(0, N-1):
        if similarity[i] < breakpoint_threshold:
            断点索引.add(i)

Step 4: 断点间组装 Chunk
    按断点索引将句子序列分组
    短组（< min_chunk_tokens）→ 与相邻组合并（选择相似度更高的一侧）
    长组（> max_chunk_tokens）→ 在该组内找到次高相似度谷底再切分

Step 5: Chunk 边界优化
    对每个候选边界，比较:
    - 边界处相似度 vs 边界左右邻域的平均相似度
    - 若边界相似度比邻域低 > 2 个标准差 → 确认为强断点
    - 否则上移/下移到最近的强断点
```

#### 参数

| 参数 | 默认值 | 范围 | 说明 |
|------|--------|------|------|
| min_chunk_tokens | 256 | 128 ~ 512 | 合并阈值 |
| max_chunk_tokens | 1024 | 512 ~ 2048 | 再切分阈值 |
| breakpoint_threshold | 0.45 | 0.30 ~ 0.65 | 语义断点相似度阈值（越低断点越少，chunk 越大） |
| breakpoint_std_multiplier | 2.0 | 1.5 ~ 3.0 | 标准差倍数（用于确认强断点） |
| window_size | 3 | 1 ~ 5 | 邻域窗口大小（计算局部平均相似度） |

#### 选择理由

- **唯一真正语义感知的切法**: 不是猜"哪里该切"，而是让模型算"哪里语义变了"
- **复杂文档最优**: 技术博客（在"介绍"→"实现"→"总结"之间自动断点）、书籍（章节内自然话题切换处切割）
- **检索精度最高**: chunk 内语义高度一致，embedding 表达精确，检索噪声最低
- **自适应文档结构**: 不需要预知文档格式，不管段落长短是否均匀，按语义边界自适应
- **主题漂移检测**: 作者话题切换时相似度自然下降 → 天然检测到主题边界

#### 不足

- **计算成本高**: 每个句子都要 embedding → N 句文档 = N 次 API 调用，长文档成本可观
- **依赖 embedding 模型质量**: 弱模型算出的相似度不可靠，断点质量差
- **不适合短文档**: < 20 句的文档语义断点太少，基本等价于整篇一个 chunk
- **延迟高**: 串行 N 次 embedding 调用 + N-1 次相似度计算，不能和其他策略比吞吐
- **建议添加成本门控**: 文档 > 10000 字时降级到 paragraph_sliding_window

## 4. 文件类型自动兜底映射

当用户上传文件时**未指定 `chunk_strategy`**，系统按以下映射表自动选择：

| 文件类型 | 兜底策略 | 推理逻辑 |
|----------|----------|----------|
| **PDF** | `paragraph_sliding_window` | PDF 通常是排版后的正式文档，天然有段落结构；Tika 解析后保留空行分隔 |
| **DOCX** | `paragraph_sliding_window` | 同类 PDF，Word 文档段落结构更规范；若有标题样式→可升级为 `markdown_header_aware`（V2） |
| **TXT** | `recursive_char_split` | 纯文本无任何结构保证，需要最通用策略兜底 |
| **MD** | `markdown_header_aware` | Markdown 的标题层级是精确的结构信号，首选用标题感知策略 |
| **HTML** | `markdown_header_aware` | HTML 的 `<h1>`~`<h6>` 天然对应 Markdown 标题；先用 Tika 保留 heading 信息，映射为 markdown_header_aware |
| **CSV** | `fixed_size` | 表格数据行结构均匀，固定窗口切分保证每 chunk 包含完整 N 行 |
| **未识别** | `recursive_char_split` | 最坏情况兜底 — 递归字符分割保证任何文本都能切 |

### 兜底策略动态调整规则

```
IF file_type IN (PDF, DOCX)
   AND Tika 解析后检测到 >= 3 个标题（H1-H3）
   → 自动升级策略为 markdown_header_aware
   → 写入 chunk_config_json: {"auto_upgraded": true, "upgrade_reason": "detected_headers>=3"}

IF file_type = TXT
   AND 文本行数 < 100 AND 检测到大量 ?/？ 结尾行
   → 自动升级策略为 sentence_level
   → 写入 chunk_config_json: {"auto_upgraded": true, "upgrade_reason": "faq_like_pattern"}

IF file_type = CSV
   AND 行数 <= 200
   → fixed_size 的 chunk_size 自动调整为 ceil(total_rows / 10)，保证约 10 个 chunk
```

## 5. API 设计

### 5.1 上传文档时指定切片策略

```
POST /api/v1/knowledge-bases/{knowledgeId}/documents
Content-Type: multipart/form-data

参数:
  file:                 (必填) 文件
  chunk_strategy:       (可选) 切片策略: paragraph_sliding_window | fixed_size | markdown_header_aware | sentence_level | recursive_char_split | semantic
  chunk_size:           (可选) 覆盖默认 chunk_size
  chunk_overlap:        (可选) 覆盖默认 chunk_overlap

行为:
  - chunk_strategy=null → 使用当前知识库 default_chunk_strategy

```

```
上传文件
    │

        │

        │
        ├─ MD → markdown_header_aware
        │
        ├─ HTML → markdown_header_aware
        │
        ├─ CSV → fixed_size
        │
        ├─ TXT → recursive_char_split
        │   （若检测到 FAQ 模式→自动升级 sentence_level）
        │
        └─ 未知 → recursive_char_split
```

## 8. 后续演进（V2）

| 进化方向 | 说明 |
|----------|------|
| **策略编排器** | 一个文档使用多种策略切分 → RRF 融合时跨策略评分，取长补短 |
| **动态参数优化** | 根据文档长度/类型/复杂度自动调参（如用 embedding 均匀性指标评估切分质量） |
| **增量更新** | 文档局部修改后，仅对修改段落重新切分+向量化，利用 content_hash 做到精确差分 |
| **切分质量评估** | 引入切分质量评分（chunk 内语义一致性 + chunk 间区分度），反馈给策略选择器 |
| **用户自定义策略** | 允许租户上传自定义分割正则/分隔符列表 |
| **PDF 标题检测升级** | 用 Apache PDFBox 检测 PDF 字体大小/加粗 → 自动识别标题层级 → 升档为 markdown_header_aware |
| **多模态切片** | 图片/表格独立切片，生成多模态 chunk（文本+图片描述+表格结构化数据） |
